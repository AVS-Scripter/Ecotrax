# Ecotrax Backend Database Integration Plan (Optimized Architecture)

This document outlines the final scalable architecture for the Ecotrax community system. It focuses on high-performance queries, operational atomicity, and secure client-side transactions.

---

## 1. `users` (Collection)
Stores user profiles and real-time membership state.

- **Document ID**: `uid` (Firebase Auth UID)
- **Fields**:
  - `displayName`: string
  - `email`: string
  - `joinedCommunities`: string[]  **[Optimized]** 
    - Enables O(1) membership validation in Security Rules.
    - Allows instant "List My Communities" dashboard queries via the `in` operator.
  - `rank`: number | null
  - `points`: number
  - `createdAt`: timestamp

---

## 2. `communities` (Collection)
Main community data repository. Uses a metadata map to separate content from state.

- **Document ID**: Auto-generated
- **Fields**:
  - `name`: string
  - `icon`: string
  - `createdBy`: userId (UID)
  - `metadata`: map
    - `memberCount`: number (Maintained via increment/decrement during transactions)
    - `isDeleted`: boolean (For soft-delete patterns)
    - `deletedAt`: timestamp | null
  - `createdAt`: timestamp

### Subcollection: `members`
Path: `communities/{communityId}/members/{userId}`

- **Fields**:
  - `role`: "admin" | "moderator" | "member"
  - `displayName`: string (Denormalized for fast list rendering)
  - `joinedAt`: timestamp

---

## 3. `invites` (Collection)
Root-level collection for universal invite code validation.

- **Document ID**: `code` (Unique short string, e.g., "AFX79G")
- **Fields**:
  - `communityId`: string
  - `communityName`: string
  - `createdBy`: string (UID)
  - `maxUses`: number | null
  - `usedCount`: number
  - `expiresAt`: timestamp
  - `isActive`: boolean
  - `createdAt`: timestamp

---

## 4. `reports` (Collection)
Environmental incidence reports scoped to communities.

- **Document ID**: Auto-generated
- **Fields**:
  - `communityId`: string (For filtering)
  - `userId`: string
  - `issueType`: "air" | "water" | "garbage" | "noise"
  - `description`: string
  - `locationName`: string
  - `location`: GeoPoint (Optional)
  - `status`: "open" | "in_progress" | "resolved"
  - `createdAt`: timestamp

---

## 5. Security & Consistency Strategy

### Atomic Operations
All membership changes (Joining an invite, Leaving, Setting Roles) MUST be performed using **Firestore Transactions**. 
- **Join**: Atomic increment of `usedCount` + document creation in `members` + array update in `users`.
- **Leave**: Atomic decrement of `memberCount` + document deletion in `members` + array removal in `users`.

### Security Rules (RBAC)
- **Member Access**: Restricted to communities listed in the user's `joinedCommunities` array.
- **Admin Access**: Verified using the `role` field in the community's `members` subcollection.
- **State Protection**: Metadata fields like `isDeleted` or `memberCount` are guarded to prevent arbitrary client manipulation outside of verified transactional flows.

### Performance Notes
- **Dashboard Load**: The use of the `joinedCommunities` array on the User object prevents N+1 queries. We fetch precisely the community documents the user belongs to.
- **Scalability**: By using root-level `invites`, we avoid global `collectionGroup` indexes for simple invite-code lookups.
- **Persistence**: Soft-deletion in `communities` ensures structural data remains for auditing while being hidden from the active user experience.
